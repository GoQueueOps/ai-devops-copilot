RUNBOOK: Pharma Application Platform

SECTION 1: OOMKilled / Memory Issues
Service: api-server
Symptoms: Pod killed, exit code 137
Steps:
1. Check current memory limit: kubectl get pod api-server -o yaml | grep memory
2. Increase memory limit to 1Gi in deployment yaml
3. Apply: kubectl apply -f deployment.yaml
4. Monitor for 10 minutes
5. If recurring, add HPA: kubectl autoscale deployment api-server --cpu-percent=70 --min=2 --max=5
Escalation: If more than 3 OOMKills in 1 hour, page on-call

SECTION 2: PostgreSQL Connection Pool Exhausted
Symptoms: FATAL: connection pool exhausted, max 20 connections
Steps:
1. Check active connections: SELECT count(*) FROM pg_stat_activity;
2. Kill idle connections: SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'idle';
3. Check application for missing db.close() calls
4. Consider PgBouncer for connection pooling
Escalation: Notify DBA team immediately via Slack #dba-alerts

SECTION 3: CrashLoopBackOff
Symptoms: Pod restarting repeatedly
Steps:
1. Check exit code: kubectl describe pod <pod-name>
2. Exit 137 = OOM kill, follow Section 1
3. Exit 1 = Application error, check application logs
4. kubectl logs <pod-name> --previous
5. Fix underlying issue before restarting pod
Escalation: If payment-service, page on-call immediately

SECTION 4: payment-service failures
ANY failure on payment-service = CRITICAL
Steps:
1. Immediately page on-call via PagerDuty
2. Check pod status: kubectl get pods -n pharma-prod
3. Check logs: kubectl logs payment-service-xxx --previous
4. Do NOT restart without on-call approval
5. Notify business stakeholders within 15 minutes
